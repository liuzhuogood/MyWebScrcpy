# 设计文档

## 背景

MyWebScrcpy 旨在提供一个零客户端安装的 Android 屏幕镜像和远程控制解决方案。传统的 scrcpy 工具需要安装客户端应用程序，而 MyWebScrcpy 通过浏览器即可实现所有功能，大大降低了使用门槛。

### 问题陈述
- 现有 scrcpy 工具需要安装客户端，增加了部署复杂度
- 跨平台支持需要为不同操作系统构建不同版本
- 远程访问需要额外的网络配置
- 移动设备上的使用体验不佳

### 解决方案
利用现代浏览器的 WebCodecs API 和 WebSocket 技术，构建一个完全基于浏览器的解决方案：
- Go 后端作为 ADB 和 scrcpy-server 的桥梁
- 浏览器前端使用 WebCodecs 进行硬件加速视频解码
- WebSocket 提供低延迟的双向通信
- 单二进制部署，内嵌所有依赖

## 目标

### 主要目标
1. **零安装体验**：用户只需打开浏览器即可使用
2. **完整功能**：提供与原生 scrcpy 相当的镜像和控制功能
3. **跨平台支持**：支持所有现代浏览器（Chrome、Firefox、Safari、Edge）
4. **移动优化**：在移动设备上提供良好的触摸交互体验
5. **单二进制部署**：简化部署和分发

### 非目标
1. **高性能游戏流**：当前配置优化为办公和演示场景（15 FPS，2 Mbps）
2. **音频传输**：当前版本不支持音频传输
3. **多设备同时镜像**：每次只支持一个设备的镜像会话
4. **企业级安全**：不提供认证、加密或访问控制
5. **录制功能**：不支持屏幕录制或截图保存

## 关键决策

### 1. 技术栈选择
**决策**：Go 后端 + 原生 JavaScript 前端
**理由**：
- Go 提供优秀的并发性能和简单的部署（单二进制）
- 原生 JavaScript 避免框架依赖，减小前端体积
- WebCodecs API 提供硬件加速解码，性能优于 Canvas 软解码
**替代方案**：
- Node.js 后端：部署复杂度高，需要运行时环境
- React/Vue 前端：增加构建复杂度和包体积
- WebAssembly 解码：兼容性和性能不如 WebCodecs

### 2. 协议实现
**决策**：实现 scrcpy 4.0 协议
**理由**：
- scrcpy 4.0 提供了现代化的协议设计
- 支持多种编解码器（H.264、H.265、AV1）
- 提供完整的控制消息支持
**替代方案**：
- 自定义协议：开发成本高，兼容性差
- scrcpy 旧版本协议：功能有限，性能较差

### 3. 视频解码策略
**决策**：使用 WebCodecs API 进行硬件加速解码
**理由**：
- 现代浏览器原生支持，性能优秀
- 支持多种编解码器，适应不同设备
- 自动利用 GPU 加速，降低 CPU 负载
**替代方案**：
- Canvas 软解码：性能差，CPU 占用高
- WebAssembly 解码：兼容性问题，部署复杂

### 4. 连接管理策略
**决策**：自动重连 + 指数退避
**理由**：
- 网络不稳定时自动恢复，提升用户体验
- 指数退避避免服务器过载
- 最大重试次数限制防止无限循环
**替代方案**：
- 手动重连：用户体验差
- 固定间隔重试：可能导致服务器过载

### 5. 部署策略
**决策**：单二进制部署，内嵌所有资源
**理由**：
- 极简部署，无需安装依赖
- 内嵌 scrcpy-server JAR 和前端资源
- 支持环境变量配置，适应不同环境
**替代方案**：
- 分离部署：增加部署复杂度
- 容器化部署：增加运行时依赖

## 风险与权衡

### 技术风险
1. **浏览器兼容性**：WebCodecs API 在旧版浏览器中可能不支持
   - 缓解：提供降级方案或明确浏览器要求
2. **性能瓶颈**：网络延迟可能影响实时性
   - 缓解：优化网络传输，提供配置参数调整
3. **安全性**：WebSocket 连接未加密
   - 缓解：在受信任网络中使用，或通过反向代理提供 TLS

### 功能权衡
1. **音频支持**：为简化实现暂不支持音频
   - 影响：某些场景（如视频通话）体验不完整
   - 后续：可在 v1.1.0 中添加音频支持
2. **多设备支持**：当前只支持单设备镜像
   - 影响：多设备管理需要多个浏览器标签页
   - 后续：可在 v1.2.0 中添加多设备管理界面
3. **录制功能**：暂不支持屏幕录制
   - 影响：用户需要使用其他工具进行录制
   - 后续：可在 v1.1.0 中添加录制功能

## 迁移/回滚计划

### 首次部署
- 无迁移需求，直接部署新版本
- 确保 ADB 已正确安装和配置
- 测试设备连接和基本功能

### 版本升级
- 停止当前服务
- 替换二进制文件
- 重启服务
- 验证功能正常

### 回滚计划
- 保留旧版本二进制文件
- 如遇问题，替换回旧版本并重启
- 检查日志定位问题
